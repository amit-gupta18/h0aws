# Billing Feature — plan.md
**Feature:** Invoice / Billing  
**Last Updated:** June 2026  
**Status:** Ready for implementation

---

## 1. Overview

The billing feature is the core of Rakhat. It allows OWNER and ACCOUNTANT roles to create GST-compliant invoices. A sale is captured, GST logic is applied, an invoice number is assigned by the server, a PDF is generated and stored on R2, and the invoice record is saved to the DB.

Invoices are immutable once created — GST compliance does not allow editing. Cancellation is the only allowed state change.

---

## 2. Roles

| Role | Create Invoice | View Invoice | Cancel Invoice |
|---|---|---|---|
| OWNER | ✅ | ✅ | ✅ |
| ACCOUNTANT | ✅ | ✅ | ✅ |
| VIEWER | ❌ | ✅ | ❌ |

Enforced via `requireRole('OWNER', 'ACCOUNTANT')` middleware on Express.

### 4.1 Create Invoice
```
POST /invoices
Roles: OWNER, ACCOUNTANT
```

**Request body:**
```typescript
{
  clientBillId: string          // UUID v4 generated by client — idempotency key
  customerId?: string           // existing customer ID, or null for walk-in
  invoiceDate: string           // ISO date: "2026-06-22"
  paymentMode: "CASH" | "UPI" | "CARD" | "CREDIT"
  notes?: string
  items: {
    productId?: string          // existing product ID, or null for ad-hoc item
    name: string                // used as nameSnapshot
    hsn?: string
    unit?: string
    quantity: number
    unitPrice: number
    discount?: number           // default 0
    gstRate: number             // 0 | 5 | 12 | 18 | 28
  }[]
}
```

**Response `201`:**
```typescript
{
  id: string
  invoiceNumber: string         // assigned by server e.g. "INV-0001"
  documentType: string
  transactionType: string
  subtotal: number
  discountTotal: number
  taxableAmount: number
  cgstTotal: number
  sgstTotal: number
  igstTotal: number
  grandTotal: number
  status: "ISSUED"
  pdfUrl: string | null         // null until PDF generation completes
  createdAt: string
}
```

**Server flow:**
```
1. authenticate → businessContext → requireRole(OWNER, ACCOUNTANT)
2. Check clientBillId — if already exists, return existing invoice (idempotent)
3. Run GST engine → get documentType, transactionType, line calculations, summary
4. Atomically increment InvoiceSequence → get invoiceNumber
5. prisma.$transaction():
     - Insert Invoice row
     - Insert SaleItem rows
6. Generate PDF via @react-pdf/renderer → upload buffer to R2
7. Update invoice.pdfUrl with R2 URL
8. Return 201 with complete invoice including pdfUrl
```

> **V1 decision:** PDF generation is synchronous — the 201 response blocks until the PDF is ready and pdfUrl is populated. Expected latency ~1-2s which is acceptable at counter. No worker queue needed. If PDF generation fails, the invoice is still saved — pdfUrl stays null and can be regenerated via `GET /invoices/:id/pdf`.

### 4.4 Get / Generate PDF
```
GET /invoices/:id/pdf
Roles: OWNER, ACCOUNTANT, VIEWER
```

**Flow:**
```
If invoice.status === CANCELLED → return 404 { error: "Invoice is cancelled" }
Check R2: does invoices/{id}.pdf exist?
  YES → return presigned URL (15 min expiry)
  NO  → generate PDF → upload to R2 → update invoice.pdfUrl → return presigned URL
```

**Response `200`:**
```typescript
{
  url: string   // presigned R2 URL, valid for 15 minutes
}
```

**Response `404`:** if invoice is CANCELLED

---

### 4.5 Cancel Invoice
```
PATCH /invoices/:id/cancel
Roles: OWNER, ACCOUNTANT
```

**Flow:**
- Set `invoice.status = CANCELLED`
- Invoice number is NOT reused — gap is intentional (GST compliance)
- Delete R2 PDF object if exists

**Response `200`:**
```typescript
{
  id: string
  status: "CANCELLED"
}
```

## 5. GST Engine

Pure TypeScript function. No DB calls. No side effects. Lives in `server/src/lib/gst-engine.ts` and duplicated in `client/src/lib/gst-engine.ts` for live preview.

```typescript
type GSTInput = {
  sellerGSTIN: string | null
  sellerStateCode: string
  buyerStateCode: string | null   // null = walk-in → treated as intra-state
  items: {
    name: string
    quantity: number
    unitPrice: number
    discount: number
    gstRate: number               // 0 | 5 | 12 | 18 | 28
  }[]
}

type GSTOutput = {
  documentType: 'TAX_INVOICE' | 'BILL_OF_SUPPLY'
  transactionType: 'INTRA_STATE' | 'INTER_STATE'
  lines: {
    taxableValue: number
    cgstAmount: number
    sgstAmount: number
    igstAmount: number
    lineTotal: number
  }[]
  summary: {
    subtotal: number
    discountTotal: number
    taxableAmount: number
    cgstTotal: number
    sgstTotal: number
    igstTotal: number
    grandTotal: number
  }
}
```

**Decision tree:**
```
Is sellerGSTIN present?
  NO  → documentType = BILL_OF_SUPPLY, all tax amounts = 0
  YES →
    Are ALL items at gstRate = 0?
      YES → documentType = BILL_OF_SUPPLY, all tax amounts = 0
      NO  → documentType = TAX_INVOICE
            Is sellerStateCode === buyerStateCode (or buyerStateCode is null)?
              YES → INTRA_STATE → CGST = rate/2, SGST = rate/2
              NO  → INTER_STATE → IGST = full rate
```

**Per line calculation:**
```
taxableValue = (quantity × unitPrice) - discount
cgstAmount   = taxableValue × (gstRate / 2) / 100   [intra-state only]
sgstAmount   = taxableValue × (gstRate / 2) / 100   [intra-state only]
igstAmount   = taxableValue × gstRate / 100          [inter-state only]
lineTotal    = taxableValue + cgstAmount + sgstAmount + igstAmount
```

All calculations use `Decimal.js` — never native JS floats.

## 6. Invoice Number Generation

Atomic increment via raw Postgres query — no race conditions.

```typescript
// server/src/invoices/invoice.service.ts
const result = await prisma.$queryRaw<{ invoice_number: string }[]>`
  UPDATE invoice_sequences
  SET current_val = current_val + 1
  WHERE business_id = ${businessId}
  RETURNING prefix || LPAD(current_val::TEXT, 4, '0') AS invoice_number
`
const invoiceNumber = result[0].invoice_number  // e.g. "INV-0001"
```

Rules:
- Number assigned by server only — never by client
- Cancelled invoices leave a gap — intentional, GST compliant
- `InvoiceSequence` row is created when business is created (in `createWithOwner` transaction)
- **Prefix must always end with `-`** — enforced at onboarding with Zod: `z.string().regex(/^[A-Z0-9]+-$/)`. If user enters `"INV"`, normalize to `"INV-"` before storing. This prevents `"INV0001"` instead of `"INV-0001"` from the raw SQL concat.

**Template notes for implementation:**
- Title: `TAX INVOICE` or `BILL OF SUPPLY` (from `documentType`) — centered, bold, large
- "ORIGINAL FOR RECIPIENT" — small text, top-right corner
- Sender logo: if `business.logoUrl` present, render as image (~48px height) immediately before the business name — i.e. the first element inside the left header block, with the name/address/GSTIN flowing below it; skip the image element entirely if null
- Business name formatted as: `M/S {tradeName}` — all caps
- Right header grid: 3 rows × 2 cols — Invoice No/Dated, Delivery Note/Mode of Payment (show `paymentMode`), Reference No./Other References (leave blank V1)
- Buyer grid (right side): 4 rows — Buyer's Order (blank), Dispatch (blank), Dispatched Through/Destination (blank), Terms of Delivery (blank) — structural placeholders
- Items table: render items + enough blank rows to reach a minimum page height (pad to ~15 rows total)
- Totals column (right of items): Subtotal | CGST | SGST (or IGST for inter-state) | Grand Total with ₹ symbol
- Bottom-left of items table: total quantity (sum of all item quantities with unit if uniform)
- Amount in words: full row spanning table width
- GST summary table: grouped by HSN/SAC — one row per unique HSN, totals row
- Tax amount in words: full row spanning table width
- Bank details: all shown as blank dashes for V1 (schema fields don't exist yet)
- "for [BUSINESS NAME]" + "Authorised Signatory" on right
- Declaration text + "This is a Computer Generated Invoice" footer

## 11. Client-side Flow

```
User opens /invoices/new
  → blank invoice form
  → clientBillId = uuid() generated immediately on mount

User types customer name
  → GET /customers?search=q  (TanStack Query, debounced 300ms)
  → shows dropdown of existing customers
  → if not found → "Add new customer" option → POST /customers inline

User adds items
  → GET /products?search=q  (TanStack Query, debounced 300ms)
  → shows dropdown of existing products
  → selecting product autofills name, HSN, unit, price, gstRate
  → if not found → "Add new item" option → POST /products inline
  → GST engine runs client-side on every item change → live preview of totals

User hits "Create Invoice"
  → POST /invoices with clientBillId
  → server blocks until PDF is generated and uploaded to R2 (~1-2s)
  → server returns 201 with invoiceNumber + pdfUrl populated
  → redirect to /invoices/:id
  → user can download PDF or share manually
```

## 12. Tradeoffs & Rejected Options

| Decision | Chosen | Rejected | Reason |
|---|---|---|---|
| PDF generation | Server-side `@react-pdf/renderer` | Client-side generation + upload | Client upload is unverifiable — server controls what goes to R2 |
| PDF timing | Generated at invoice creation | Generated on first download | Shopkeeper should not wait at counter for PDF |
| PDF storage | Cloudflare R2 | Regenerate every time | Regenerating on every download wastes compute |
| Invoice number | Server-assigned atomic increment | Client-assigned | Two simultaneous bills would collide |
| Invoice mutability | Immutable after creation | Allow edits | GST compliance — issued invoices cannot be altered |
| Cancelled invoices | Status = CANCELLED, number kept | Hard delete | GST audit trail requires gap in sequence, not missing records |
| Template sharing | Duplicated in client + server | Shared npm package | Shared package is overkill for V1; template rarely changes |
| Worker queue for PDF | Not used in V1 | BullMQ + Redis queue | V1 scale does not justify it; inline generation is fine |
